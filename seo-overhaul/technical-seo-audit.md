# Technical SEO Audit & Backend Overhaul — thailotterynumbers.com

**Scope:** Backend only. No changes to visible page content, copy, images, layout, UI, tools, or statistics displays. Every recommendation lives in the HTML `<head>`, server/Netlify config (`_headers`, `_redirects`, `netlify.toml`), `robots.txt`, `sitemap.xml`, HTTP response headers, or JSON-LD.
**Canonical host:** `https://thailotterynumbers.com` (non-www). **Date:** 2026-07-06.
**Stack observed:** Static multipage site pre-rendered by `build_seo.py` (one real URL/`index.html` per route) + client-side hydration via `assets/app.js`, deployed on Netlify. `build_seo.py` also generates `robots.txt`, `sitemap.xml`, `_redirects`, and `404.html`.

> **Persistence note (read first):** `build_seo.py` regenerates `robots.txt`, `sitemap.xml`, `_redirects`, `404.html`, and every page `<head>` on each run. Dropping the new files into the site root fixes production **now**, but they will be overwritten on the next `python build_seo.py`. Each section below includes a **"Make it durable in `build_seo.py`"** note so the changes survive rebuilds. `_headers` is *not* generated by the script, so it is a clean, permanent new file.

## 1. Executive Summary

The site is already in good technical shape: unique per-page `<title>`/description/canonical, an `@graph` JSON-LD (Organization + WebSite + WebPage + BreadcrumbList), FAQ schema on `/faq`, a sitemap, a robots file with some AI-bot handling, `ads.txt`, pre-rendered crawlable intro copy on every route, and `display=swap` fonts. This audit is therefore an **overhaul of a decent baseline**, not a rescue.

The highest-impact gaps found:

1. **Soft-404 pattern (critical).** `_redirects` uses `/*  /index.html  200`, so every unknown URL returns the **homepage with HTTP 200**. `404.html` is also a byte-for-byte copy of the homepage. Google treats these as soft 404s, which wastes crawl budget and can suppress quality signals. **Fixed** by serving `/404.html` with a real `404` status.
2. **No security/trust headers.** There is no `_headers` or `netlify.toml`, so HSTS, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, CSP, and cache-control are all unset. These are direct E-E-A-T/trust signals and CWV levers. **Fixed** with a full `_headers` file.
3. **AI-crawler policy is inconsistent with intent.** Current robots allows `Google-Extended` (a training token) while omitting the newer answer/search tokens (`Claude-SearchBot`, `Perplexity-User`, etc.) and the training tokens you'd want to block (`GPTBot`, `CCBot`, `Bytespider`…). **Rewritten** to *allow answer engines, block trainers* per your decision.
4. **`/contact` is orphaned.** The page exists with valid meta but is absent from `PAGES` in `build_seo.py`, so it never enters the sitemap. **Added.**
5. **`robots` meta is missing preview directives.** Adding `max-image-preview:large, max-snippet:-1, max-video-preview:-1` unlocks large image thumbnails and full snippets in Search and AI Overviews.
6. **No `Dataset` / `WebApplication` schema.** A results/statistics site is an ideal `Dataset` candidate (Google Dataset Search eligibility) and the checker/search tools are textbook `WebApplication` entities. **Added.**
7. **Cache-Control absent.** Fingerprint-free assets need an explicit, sane caching posture for LCP/repeat-view performance. **Added** in `_headers`.
8. **`Disallow: /cookie-settings` conflicts with its own `noindex`.** Blocking the URL in robots stops Google from ever *seeing* the `noindex`. **Removed** the disallow; rely on `noindex` + an `X-Robots-Tag` header.

Expectation-setting for 2026 (important accuracy notes): Google restricted **FAQ rich results** to authoritative government/health domains in 2023 and **deprecated HowTo rich results** — so FAQ/HowTo markup here will rarely produce classic rich cards, but it remains highly valuable for **AI Overviews, ChatGPT Search, and Perplexity** parsing, which is why we keep and extend it. `Dataset` markup mainly earns visibility in **Google Dataset Search** and AI answers rather than a mainstream SERP carousel. Recommendations are calibrated to that reality.

**Why this split works in 2026:** training and answer functions are now separate user-agents. Blocking `GPTBot`/`Google-Extended`/`ClaudeBot`/`CCBot`/`Applebot-Extended` opts you out of model-training corpora, while `OAI-SearchBot`, `Claude-SearchBot`, `PerplexityBot` (+ their `*-User` fetchers) keep you eligible to be **cited** in AI answers. Google's AI Overviews are fed by ordinary `Googlebot`, which stays fully allowed — so blocking `Google-Extended` does **not** remove you from AI Overviews. A per-agent group overrides the `*` group, so each listed bot obeys its own block/allow.

### 2.5 Optional: single-file `netlify.toml` equivalent

If you prefer one config file over `_headers` + `_redirects`, use this instead (don't use both for the same rules):

```toml

  to = "/404.html"
  status = 404

[[headers]]
  for = "/*"
  [headers.values]
```

## 4. Additional Backend Recommendations

### 4.1 Core Web Vitals — backend/config levers (no visible-content changes)

**LCP.** The homepage LCP element is the pre-rendered `<h1>`/intro, so the main risk is render-blocking web fonts loaded from Google's CDN (three families — Kanit, Sarabun, Space Mono — with many weights). Levers, all `<head>`/config:
- Keep `display=swap` (already present) so text paints immediately.
- Add `<link rel="preload" href="/assets/styles.css" as="style">` before the stylesheet (in §3.1 head).
- **Self-host the fonts** under `/assets/fonts/` and serve them with the 1-year immutable cache from `_headers`. This removes the `fonts.googleapis.com` → `fonts.gstatic.com` round-trip from the critical path, improves LCP on repeat views, and simplifies your CSP (drop the Google Fonts allowances). Subset to the weights actually used. This is a build/asset change, not a content change.
- Consider trimming font weights (e.g., 400/600/700 only) to cut transfer size — a config decision in the font `<link>`/`@font-face`, not visible copy.

**INP.** Third-party ad scripts (Adsterra `highperformanceformat.com`, plus AdSense per `ads.txt`) are the main INP/ main-thread risk. Backend-safe mitigations:
- `<link rel="preconnect" href="https://www.highperformanceformat.com">` (added in §3.1) and `dns-prefetch` for ad domains speed their handshake.
- Your first-party JS is already correct: `chart.umd.js`, `i18n.js`, `app.js` all use `defer`. Keep chart/tool code out of the critical path (it already is).

**CLS.** Ad slots declare fixed `width`/`height` (160×600, 728×90), which reserves space and prevents layout shift — good; keep those attributes. The self-hosted-font swap above also avoids late-font reflow.

**Delivery.** Netlify serves Brotli/gzip and HTTP/2/3 automatically — no action needed, but verify Brotli is active on `.css`/`.js`/`.json` (see §6). The `_headers` cache policy gives near-instant repeat views without risking stale results (HTML/data revalidate; static binaries cache hard).

**Images.** `Thailand-Chiang-Mai-Gate-Market-Lottery-Seller-lg.jpg` (~154 KB) and `og-image.png` are cached hard by `_headers`. Optional build-step win: emit AVIF/WebP variants and reference them via `<picture>`/`srcset` — but that touches markup, so treat it as a *future* option outside this backend-only pass. Caching alone covers repeat views now.

### 4.2 Security & trust headers (E-E-A-T)

The `_headers` file (§2.3) sets:

- **`Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`** — forces HTTPS for two years across subdomains and makes you eligible for the browser HSTS preload list. Only keep `preload`/`includeSubDomains` if *all* subdomains are HTTPS-ready (they are for a Netlify apex).
- **`X-Content-Type-Options: nosniff`** — blocks MIME-sniffing attacks.
- **`X-Frame-Options: SAMEORIGIN`** (+ CSP `frame-ancestors 'self'`) — anti-clickjacking; your own ad iframes are children of your page, so this doesn't affect them.
- **`Referrer-Policy: strict-origin-when-cross-origin`** — sends full referrer same-origin, only the origin cross-origin. Good privacy default.
- **`Permissions-Policy: geolocation=(), microphone=(), camera=(), payment=(), usb=(), browsing-topics=()`** — disables powerful APIs the site never uses. **Trade-off:** `browsing-topics=()` opts you out of the Topics API; if AdSense ad revenue benefits from Topics-based targeting, remove that one token. For a trust-first info site, disabling it is the cleaner default.
- **`Cross-Origin-Opener-Policy: same-origin-allow-popups`** — isolation that still allows ad/OAuth pop-ups.
- **`Cache-Control`** — see §4.4.

**CSP strategy (why Report-Only first).** Your ads use `srcdoc` iframes containing **inline `<script>`**, which inherit the parent CSP. A strict `script-src` without `'unsafe-inline'` would break them, and inline `style="…"` attributes require `'unsafe-inline'` in `style-src`. So we ship **`Content-Security-Policy-Report-Only`** to log violations without breaking anything. Review the reports for ~1 week, confirm every legitimate ad/font/analytics origin is listed, then **promote to enforcing** by renaming the header to `Content-Security-Policy`. The enforcing target (same directives, keeping `'unsafe-inline'` because of the ad `srcdoc` + inline styles):

```text
Content-Security-Policy: default-src 'self'; base-uri 'self'; object-src 'none'; frame-ancestors 'self'; script-src 'self' 'unsafe-inline' https://www.highperformanceformat.com https://*.highperformanceformat.com https://pagead2.googlesyndication.com https://*.googlesyndication.com https://adservice.google.com https://*.google.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data: https:; connect-src 'self' https:; frame-src https:; upgrade-insecure-requests
```
If you later self-host fonts (§4.1), drop the `fonts.googleapis.com`/`fonts.gstatic.com` origins. Because ad networks change origins, prefer promoting only after Report-Only is quiet.

### 4.3 Canonicalization & duplicate-content strategy

- **Host:** single canonical host `https://thailotterynumbers.com` (non-www). Set the apex as the **primary domain** in Netlify so www→apex and http→https 301s are automatic; the `_redirects` comments include explicit rules if you're off Netlify DNS. Every canonical/OG-URL/sitemap-loc/schema `@id` already uses non-www — consistent.
- **Trailing slash:** you use clean, slash-less pretty URLs (`/faq`, not `/faq/`) and canonicals match. Netlify serves `/faq` from `/faq/index.html` without adding a slash — consistent; don't introduce mixed forms.
- **Query strings:** `/search?q=` self-canonicalizes to `/search` (§3.5). Safe.
- **Per-draw duplicate risk / opportunity:** there are currently no per-date URLs, so no duplication — but also no way for a single draw to rank. **Optional high-value, content-neutral enhancement:** have `build_seo.py` emit one static page per draw (e.g., `/results/2026-06-16`) *from data you already publish*, each with a unique `<title>`/description and a `Dataset`-or-`Article`-style JSON-LD, a self-canonical, and an entry in a dedicated `sitemap-results.xml`. This exposes existing data at crawlable URLs (captures "1 June 2026 Thai lottery result" long-tail) without adding new editorial content. Treat as a phase-2 project; flagged here because it's the single biggest untapped organic lever for a results site.

### 4.4 Caching & crawl-budget

- **Cache-Control** (from `_headers`): HTML and `/assets/data/*` revalidate (fresh results fast); fonts/images/`chart.umd.js` cache for a year; unfingerprinted `app.js`/`styles.css` revalidate hourly with `stale-while-revalidate`. If you add content-hashing to asset filenames later, switch those to `immutable`.
- **Crawl budget** is a non-issue at ~25 URLs — the levers that matter are the ones already addressed: kill soft-404s (§2.4), keep the sitemap clean and current, and avoid parameter sprawl (canonicals handle `?q=`). If you ship per-draw pages (§4.3), split the sitemap into a **sitemap index** (`sitemap.xml` → references `sitemap-core.xml` + `sitemap-results.xml`) to stay under the 50k-URL / 50 MB limits and to give GSC per-section coverage reporting.
- **`X-Robots-Tag`** is used only where needed: `noindex` on `/cookie-settings`. Genuine 404s return a `404` status (no indexing possible), so no tag needed there.

### 4.6 Bilingual (English / Thai) handling

Today both languages live at the **same URL**, swapped client-side by `i18n.js`/`localStorage`; the *pre-rendered* HTML (the part crawlers read without executing the toggle) is **English**. Implications and backend-only handling:
- **hreflang is not applicable** with one URL per two languages — hreflang requires distinct URLs (`/` vs `/th/…`). Don't add self-referential or fabricated hreflang; it would be invalid.
- **Keep** `og:locale` = `en_US` with `og:locale:alternate` = `th_TH` (present) and `inLanguage: ["en","th"]` on WebSite / `"en"` on each WebPage (present). Optionally add a `Content-Language: en` response header for the default.
- **Reality to accept:** because Thai copy is injected client-side and isn't in the pre-rendered HTML, Thai-language queries will mostly match your English pages via Google's rendering rather than ranking as native Thai content. **The only way to fully win Thai SEO is separate, server-rendered Thai URLs** (e.g., `/th/…`) with their own `<title>`/meta/hreflang — that's a build/architecture change and touches rendered text, so it sits outside this backend-only pass. Flagged as the biggest bilingual opportunity; implement when you're ready to generate a Thai URL tree from your existing translations (no *new* content required, just exposing the Thai you already have at crawlable URLs + reciprocal hreflang).
